# Question 9

*UGC NET CS · 2013 Dec Paper 3 · Software Project Management · Software Myths and Outsourcing Governance*

Which once of the following is not a software myth ?

- **A.** Once we write the program and get it to work, our job is done.
- **B.** Project requirements continually change, but change can be easily accommodated because software is flexible.
- **C.** If we get behind schedule, we can add more programmers and catch up.
- **D.** If an organization does not understand how to control software projects internally, it will invariably struggle when it outsources software projects.

> [!TIP]
> **Correct answer: D. If an organization does not understand how to control software projects internally, it will invariably struggle when it outsources software projects.**

## Solution

D is a realistic management principle: an organization that cannot govern internal software work will not automatically gain control by outsourcing it. Contracts do not replace requirements clarity, monitoring and technical governance. The other three are classic software myths.

## Key Points

- Outsourcing transfers implementation work, not the need for competent project governance.

## Why the other options are incorrect

A ignores maintenance, B underestimates the cost and ripple effects of change, and C conflicts with Brooks's law: adding people to a late project can make it later because of training and communication overhead.
